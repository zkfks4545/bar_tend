# Re:Station 프로젝트 방향성

## 한 줄 소개

> 카루아와 짧게 대화하고, 가끔 시에스타의 만담을 마주치며 지금의 기분과 취향에 어울리는 한 잔을 추천받는 가상의 바

## 현재 제품 목표

Re:Station의 현재 구현 목표는 게임 전체가 아니라 **대화 + 칵테일 추천 웹 서비스 MVP**다.

사용자는 가상의 바 Re:Station에 방문한 손님처럼 바텐더 알바생 **카루아**와 짧은 대화를 나눈다. 카루아는 사용자의 기분, 상황, 취향을 파악하고 필요한 경우 1~3개의 추가 질문을 한 뒤 어울리는 칵테일을 추천한다.

서비스는 상담소가 아니다. 사용자의 문제를 해결하거나 직접 위로하기보다 농담과 한 잔의 추천으로 잠시 분위기를 환기한다.

## MVP 범위

### 반드시 구현

- 카루아의 초기 환영 메시지
- 자유 입력 기반 채팅
- 카루아의 짧은 반응과 필요한 경우 1~3개의 추가 질문
- DB 기반 칵테일 추천
- 추천 결과 카드
- 다시 추천받기
- 모바일 사용 가능
- 규칙 기반 응답 복구 경로
- JSON·규칙 기반 카루아 대화와 추천 설명
- 낮은 빈도의 시에스타·카루아·손님 만담 이벤트

### 선택 구현

- 무알코올 추천 토글
- 도수 강도 선택
- 단맛, 쓴맛, 상큼함 선호 선택
- 대화 로그 초기화
- 기존 도감과 BGM 유지

### MVP에서 제외

- 시에스타를 상시 대화 캐릭터로 선택하는 기능
- 웰컴 드링크 랜덤 이벤트
- Restination 과거 서사의 본격 전개
- 재고, 주문, 경영, 타이쿤 시스템
- 사용자 계정과 서버 동기화
- 백엔드 기반 AI 호출
- 제휴 바 확장과 O2O 기능 고도화

## 핵심 가치

| 가치 | 정의 | 검수 질문 |
|---|---|---|
| 환기 | 고민을 해결하지 않지만 고민만 바라보게 두지도 않는다. | 대화 후 잠시 분위기가 전환되는가? |
| 캐릭터 대화 | 기능 안내가 아니라 카루아와 대화하는 감각을 제공한다. | 응답에서 카루아의 성격과 말투가 유지되는가? |
| 취향 탐색 | 사용자가 자신의 기분과 취향에 맞는 칵테일을 발견한다. | 추천 결과와 이유가 사용자 입력에 연결되는가? |
| 짧고 가벼운 사용 | 복잡한 절차 없이 한 번의 방문 경험을 완성한다. | 첫 사용자가 설명 없이 대화와 추천을 완료할 수 있는가? |

## 대화 안전 경계

카루아는 사용자의 감정을 인식할 수 있지만 상담가, 치료자, 구원자처럼 행동하지 않는다.

### 해야 하는 것

- 사용자의 표현을 짧게 받아준다.
- 가벼운 농담으로 분위기를 전환한다.
- 추천에 필요한 취향을 자연스럽게 질문한다.
- 위험하거나 심각한 상황에서는 농담을 멈추고 안전한 안내를 제공한다.

### 하지 않는 것

- 문제를 진단하거나 해결책을 단정한다.
- 과도한 위로나 감정적 의존을 유도한다.
- 사용자를 모욕하거나 고통을 가볍게 소비한다.
- 음주가 문제 해결책인 것처럼 표현한다.
- 확인되지 않은 칵테일 정보를 생성한다.

## 서비스 흐름

1. 사용자가 Re:Station에 접속한다.
2. 카루아가 초기 환영 메시지를 보낸다.
3. 사용자가 현재 상황, 기분 또는 취향을 자유롭게 입력한다.
4. 카루아가 짧게 반응한다.
5. 추천에 정보가 부족하면 현재 상태와 직전 답변에 맞는 1~3개의 추가 질문을 이어간다.
6. 추천 엔진이 사용자 신호와 칵테일 DB를 기반으로 결과와 추천 근거를 결정한다.
7. 카루아가 결정된 추천 근거를 자신의 말투로 표현한다.
8. 추천 카드가 표시된다.
9. 사용자는 다시 추천받거나 대화를 이어갈 수 있다.
10. 추천 흐름을 방해하지 않는 구간에는 시에스타가 잠깐 등장해 카루아와 손님 사이의 짧은 만담 이벤트가 발생할 수 있다.

## 시에스타 만담 이벤트

시에스타는 카루아와 동등한 상시 대화 캐릭터가 아니다. 손님과 카루아가 대화하던 중 원래 하던 바 업무를 잠깐 멈추고 불쑥 끼어들며, 짧은 만담 뒤에는 창고 정리나 청소 같은 일을 하러 다시 빠진다.

| 항목 | 원칙 |
|---|---|
| 등장 빈도 | 낮게 유지하며 한 세션에서 반복 노출을 제한 |
| 길이 | 일반적으로 2~4개의 짧은 발화 |
| 참여자 | 시에스타, 카루아, 손님 중 최소 둘 이상 |
| 목적 | 관계성, 만담, 분위기 환기 |
| 등장 방식 | 예고나 정식 인사 없이 진행 중인 대화에 불쑥 난입 |
| 대화 대상 | 카루아 또는 손님 한쪽과 먼저 만담하고 필요하면 삼자 대화로 확장 |
| 퇴장 방식 | 창고 정리, 청소, 재고 확인, 잔 정리 등 바 업무로 복귀 |
| 추천 관여 | 추천 결과 선택 및 변경 금지 |
| 발생 구간 | 일반 대화 중간, 추천 완료 직후, 재방문 분위기 |
| 발생 금지 | 추천 질문 진행 중, 안전 확인 중, 퇴장 및 초기화 중 |

이벤트는 사용자의 입력을 무시하는 랜덤 대사 묶음이 아니라, 직전 대화의 주제나 추천 결과를 짧게 받아 만담으로 연결한다. 시에스타의 난입은 그 순간을 위해 기다리고 있던 등장처럼 보이면 안 된다. 사용자가 시에스타에게 직접 말을 걸면 이벤트 안에서 짧게 답할 수 있지만, 시에스타는 업무 복귀를 이유로 퇴장하고 이후 대화의 중심은 자연스럽게 카루아에게 돌아온다.

## 추천 출력 계약

추천 결과는 다음 정보를 포함한다.

| 항목 | 책임 |
|---|---|
| 칵테일 이름 | DB |
| 간단한 설명 | DB |
| 추천 이유 | 추천 엔진이 근거 결정, 규칙 엔진이 대화창에서 카루아 말투로 표현 |
| 맛 프로필 | DB |
| 알코올 도수 느낌 | DB |
| 카루아식 한마디 | 규칙 엔진이 대화창에 표시 |

추천 카드는 이름, 이미지, 분위기 요약, DB 기반의 중립적인 설명, 베이스, 재료, 잔, 분류와 맛 프로필을 표시한다. 추천 이유, 농담, 카루아식 한마디는 카드에 넣지 않고 대화창에 남긴다. 상세 레시피는 사이드바의 레시피 패널에서 제공한다.

## 역할 분리 원칙

| 영역 | 담당 역할 |
|---|---|
| 칵테일 DB | 이름, 설명, 맛, 향, 도수감, 재료, 태그의 기준 정보 제공 |
| 추천 엔진 | 사용자 입력을 구조화하고 추천 결과와 근거 결정 |
| WebLLM | 일반 대화와 이미 결정된 추천 근거를 카루아식 문장으로 표현 |
| 규칙 엔진 | WebLLM 미지원, 미준비, 오류 시 대화와 추천 설명 제공 |
| UI | 채팅, 상태, 추천 카드, 선택 입력을 사용자에게 전달 |

### 추천 질문 단계

| 단계 | 동작 |
|---|---|
| WebLLM 도입 전 | JSON으로 정의된 질문 주제, 선택지, 상태 반영 규칙과 카루아식 문구를 사용한다. |
| WebLLM 재개 후 | JSON·규칙 로직이 확정한 답안을 WebLLM이 카루아 말투로만 포장한다. |

질문은 고정 순서 설문으로 진행하지 않는다. 직전 답변을 짧게 받아치고, 이미 확인한 항목은 반복하지 않으며, 추천에 필요한 정보가 충분해지면 즉시 종료한다. 선택 버튼은 빠른 답변을 돕는 보조 수단이며 자유 입력을 항상 허용한다.

질문 축은 맛과 풍미, 도수, 탄산, 베이스 기주로 제한한다. 맛과 풍미를 우선해 초심자도 답하기 쉽게 하고, 롱·숏 드링크는 검수된 서빙 스타일 데이터가 추가되기 전까지 별도 질문으로 사용하지 않는다.

`잘 모르겠어요`는 현재 질문만 건너뛰고 필요한 경우 다음 질문을 이어간다. `카루아에게 맡기기`는 추가 질문을 종료하고 이전까지 수집한 기분, 상황, 취향과 제약을 반영해 즉시 추천한다.

### 절대 규칙

- 입력에 등록된 칵테일 이름 또는 별칭이 있으면 취향 기반 추천보다 이름 검색을 우선한다.
- 이름 검색은 이름과 별칭만 사용하며 재료, 향, 베이스 표현을 특정 칵테일 이름으로 오인하지 않는다.
- WebLLM은 추천 칵테일을 선택하거나 변경하지 않는다.
- WebLLM은 사용자 입력이나 추천 상태를 해석·추출·변경하지 않는다.
- WebLLM은 DB에 없는 칵테일 정보나 추천 근거를 만들지 않는다.
- 추천 엔진은 WebLLM 없이도 완전히 동작해야 한다.

## WebLLM 방향

WebLLM 구현은 잠정 보류한다. 재개 시에도 JSON·DB·규칙 로직이 확정한 답안의 말투를 포장하는 선택적 후처리 계층으로만 사용한다.

| 항목 | 결정 |
|---|---|
| 실행 위치 | 브라우저 내부, Web Worker |
| 우선 모델 후보 | Qwen 계열 |
| 비교 모델 후보 | Gemma 계열 |
| 선정 기준 | 한국어 자연스러움, 카루아 말투 유지, 짧은 응답, 지시 준수, 속도와 메모리 |
| 모델 미지원 시 | 규칙 기반 응답으로 복구 |
| 모델 다운로드 | 사용자가 명시적으로 시작하며 진행률과 예상 부담을 표시 |
| 추천 관여 | 금지 |

모델은 이름만으로 확정하지 않는다. 실제 WebLLM 지원 여부와 Re:Station 전용 한국어 평가 세트 결과로 최종 선택한다.

### 체감 응답 속도 전략

1. 사용자 입력 직후 100ms 안에 카루아의 표정, 타이핑 상태 또는 짧은 규칙 반응을 표시한다.
2. 규칙 엔진이 즉시 사용할 수 있는 첫 문장 또는 추천 결과를 먼저 제공한다.
3. WebLLM은 Web Worker에서 백그라운드 실행하고 첫 토큰부터 스트리밍한다.
4. 추천 카드와 DB 기반 결과는 WebLLM을 기다리지 않고 즉시 표시한다.
5. 출력은 기본 1~3문장으로 제한하고 불필요한 대화 기록과 프롬프트 토큰을 줄인다.
6. 모델은 사용자가 대화 화면에 진입한 뒤 유휴 시간에 준비하되, 최초 다운로드는 사용자 동의 후 시작한다.
7. 모델과 KV 캐시는 세션 동안 유지하고 매 응답마다 재초기화하지 않는다.
8. 시간 예산을 넘으면 생성 요청을 중단하고 이미 표시한 규칙 응답을 최종 응답으로 유지한다.

### 측정 지표

| 지표 | 목표 |
|---|---|
| 입력 후 첫 시각 반응 | 100ms 이내 |
| 준비된 모델의 첫 토큰 | 800ms 이내 목표 |
| 추천 결과 카드 표시 | WebLLM 대기 없이 즉시 |
| 일반 생성 길이 | 1~3문장 |
| 시간 초과 시 동작 | 규칙 응답 유지, UI 대기 해제 |

## UI 방향

### 화면 구성

- 중앙 채팅 영역
- 하단 입력창
- 카루아 대사는 왼쪽
- 사용자 대사는 오른쪽
- 추천 결과는 카드 형태
- 선택 질문은 버튼과 자유 입력을 함께 지원

### 시각 방향

| 영역 | 방향 |
|---|---|
| 전체 분위기 | 어두운 바, 부드러운 네온, 과하게 화려하지 않음 |
| 배경 | 어두운 남색 또는 흑회색 |
| 포인트 | 연보라 |
| 텍스트 | 밝은 회색 |
| 추천 카드 | 반투명한 어두운 패널 |
| 반응형 | 모바일에서 채팅, 입력, 추천 카드가 가려지지 않음 |

## 기능 추가 기준

- [ ] 카루아와 대화하는 감각 또는 칵테일 추천 경험을 직접 강화한다.
- [ ] MVP의 짧고 가벼운 흐름을 방해하지 않는다.
- [ ] 추천 결과를 바꾸는 기능이라면 DB 기반 근거가 있다.
- [ ] WebLLM 없이도 핵심 기능이 동작한다.
- [ ] 모바일 사용성과 접근성을 해치지 않는다.
- [ ] 게임 확장이나 세계관 설명을 MVP에 과도하게 끌어오지 않는다.

## MVP 성공 기준

- [ ] 첫 사용자가 설명 없이 카루아와 대화를 시작할 수 있다.
- [ ] 카루아가 1~3개의 질문 안에서 추천에 필요한 정보를 얻는다.
- [ ] 추천 결과가 사용자 기분, 상황 또는 취향과 연결된다.
- [ ] 추천 카드에 기존 상세 정보와 중립 설명이 표시되고 카루아식 추천 멘트는 대화창에 표시된다.
- [ ] 다시 추천받기가 정상 동작한다.
- [ ] WebLLM 미지원 또는 실패 시에도 전체 추천 흐름을 완료할 수 있다.
- [ ] 모바일에서 핵심 흐름을 완료할 수 있다.
- [ ] 사용자가 상담받았다는 느낌보다 잠시 쉬어갔다는 느낌을 받는다.

- 도감과 BGM 고도화
