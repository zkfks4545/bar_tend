# AI 작업자 운영 규칙

## 목적

이 문서는 교체 가능한 작업자가 일관된 방식으로 작업을 시작하고, 수행하고, 종료하도록 돕는 공통 운영 규칙이다. 작업자는 특정 도구나 모델에 종속되지 않으며, 모든 판단과 변경은 확인 가능한 근거와 작업 기록을 남겨야 한다.

## 작업 시작 절차

아래 순서를 지킨다.

1. `CURRENT_STATE.md`에서 현재 상태와 우선순위를 확인한다.
2. `TASK_BOARD.md`에서 수행할 작업의 상태와 범위를 확인한다.
3. `HANDOVER.md`에서 직전 작업자의 진행 상황과 주의사항을 확인한다.
4. 구조에 영향을 주는 작업이면 `ARCHITECTURE.md`를 확인한다.
5. 기존 결정과 충돌할 가능성이 있으면 `DECISIONS.md`를 확인한다.
6. 캐릭터 대사, 프롬프트, 대화 평가 작업이면 `CHARACTER_DESIGN.md`를 확인한다.

### 시작 체크리스트

- [ ] 수행할 작업 ID가 있다.
- [ ] 작업 목적과 완료 조건을 이해했다.
- [ ] 수정 가능 범위와 수정 금지 범위를 확인했다.
- [ ] 관련 파일과 기존 변경 사항을 확인했다.
- [ ] 캐릭터 대화 작업이면 대상 캐릭터의 발화 알고리즘과 금지 패턴을 확인했다.
- [ ] 차단 요소와 알려진 문제를 확인했다.
- [ ] 필요한 검증 방법을 정했다.

## 작업 수행 규칙

| 규칙 | 기준 |
|---|---|
| 범위 준수 | 작업 범위를 벗어난 수정은 하지 않는다. 필요한 경우 별도 작업으로 등록한다. |
| 구조 보존 | 완료 조건에 필요하지 않은 구조 변경과 대규모 정리는 하지 않는다. |
| 근거 우선 | 코드, 문서, 실행 결과로 확인되지 않은 내용을 사실처럼 기록하지 않는다. |
| 변경 기록 | 설계, 데이터 구조, 외부 계약, 운영 방식 변경은 `DECISIONS.md`에 기록한다. |
| 기존 변경 보호 | 다른 작업자가 만든 변경을 임의로 삭제하거나 되돌리지 않는다. |
| 검증 수행 | 변경 범위에 맞는 빌드, 린트, 테스트, 수동 확인을 수행한다. |
| 실패 기록 | 실패한 시도와 원인을 `WORK_LOG.md`에 남긴다. |
| 작업자 식별 | `WORK_LOG.md`의 작업자 항목에는 실제 작업 중인 AI 모델명을 기록한다. 모델명을 확인할 수 없으면 추측하지 않고 `AI 모델 미기록`으로 표시한다. |

### 작업 중 체크리스트

- [ ] 현재 작업의 완료 조건에 직접 필요한 변경만 수행하고 있다.
- [ ] 새로운 가정이 생기면 확인하거나 `미확인`으로 표시했다.
- [ ] 설계 변경이 생기면 결정 기록을 작성했다.
- [ ] 새로 발견한 문제는 현재 작업에 섞지 않고 작업 후보로 기록했다.
- [ ] 사용자 데이터, 비밀 정보, 기존 동작을 불필요하게 손상하지 않았다.

## 작업 종료 절차

1. `WORK_LOG.md`에 수행 내용과 검증 결과를 기록한다.
2. `HANDOVER.md`를 다음 작업자가 즉시 이어받을 수 있게 갱신한다.
3. `TASK_BOARD.md`에서 작업 상태와 진행 상황을 갱신한다.
4. `CURRENT_STATE.md`에서 현재 상태, 우선순위, 이슈를 갱신한다.

### 종료 체크리스트

- [ ] 완료 조건을 모두 확인했다.
- [ ] 변경 파일과 생성 또는 삭제 파일을 기록했다.
- [ ] 실행한 검증과 결과를 기록했다.
- [ ] `WORK_LOG.md` 작업자 항목에 현재 AI 모델명을 기록했다.
- [ ] 실패한 검증 또는 미실행 검증을 기록했다.
- [ ] 남은 작업과 알려진 문제를 기록했다.
- [ ] 다음 작업자가 먼저 읽을 파일을 지정했다.
- [ ] 작업 상태가 실제 상태와 일치한다.

## 인수인계 규칙

인수인계는 다음 작업자가 추가 질문 없이 첫 행동을 결정할 수 있을 정도로 구체적으로 작성한다.

| 항목 | 작성 기준 |
|---|---|
| 현재 위치 | 어떤 작업 ID를 어디까지 수행했는지 기록한다. |
| 완료 내용 | 확인된 결과만 기록한다. |
| 남은 내용 | 다음에 실행할 구체적인 행동을 순서대로 기록한다. |
| 관련 파일 | 먼저 읽을 파일과 이유를 기록한다. |
| 주의사항 | 깨지기 쉬운 계약, 기존 변경, 금지 범위를 기록한다. |
| 검증 | 실행한 명령, 결과, 남은 검증을 기록한다. |
| 차단 요소 | 원인, 영향, 해제 조건을 기록한다. |

### 인수인계 품질 체크리스트

- [ ] 작업 ID와 현재 상태가 명확하다.
- [ ] 다음 행동이 동사로 시작하는 구체적인 문장이다.
- [ ] 파일 경로와 검증 방법이 포함되어 있다.
- [ ] 사실과 추측이 구분되어 있다.
- [ ] 임시 변경, 우회 처리, 위험 요소가 숨겨져 있지 않다.
- [ ] 다음 작업자가 되돌리면 안 되는 변경이 표시되어 있다.

## 작업 로그 번호 통합 규칙

- 같은 본 작업 번호의 구현, 검토, 버그 수정, 보완 작업은 `WORK_LOG.md`의 기존 항목에 누적한다.
- 단순 후속 보완을 기록하기 위해 `RST-501-A`처럼 새 접미사 작업 코드를 만들지 않는다.
- 작업 보드에서 독립 범위와 완료 조건이 정의된 하위 단계만 `RST-602-A` 같은 접미사 코드를 사용할 수 있다.
- 접미사 하위 단계가 여러 개이면 `RST-602` 통합 이력 아래에 단계별 소제목으로 모아 기록한다.
- 여러 작업 번호를 함께 수행했으면 각 본 작업 로그에 관련 변경을 나누어 기록한다.
