## DISC-001: JSON 중심 대화 계약과 미등록 칵테일 발견

| 항목 | 내용 |
|---|---|
| 상태 | MVP 이후 검토할 논의 주제 |
| 문제 | DB에 없는 칵테일 이름을 주문하면 일반 대화로 빠지거나 대화가 단절될 수 있다. 향후 LLM을 연결하더라도 전체 대화 판단을 모델에 맡기면 비용, 지연, 재현성 부담이 커진다. |
| 제안 | 사용자 입력의 의미와 다음 행동을 `DialogueTurn` JSON으로 구조화한다. 규칙 엔진 또는 LLM은 동일한 JSON 후보를 만들고, 애플리케이션이 스키마 검증 후 상태 변경과 행동을 확정한다. |
| JSON 규모 통제 | 대사 전문과 상황별 문장을 JSON에 누적하지 않는다. 공통 스키마에 `intent`, `entities`, `statePatch`, `action`, `responseGoal`, `facts`, `forbidden`, `confidence`만 담고, 기본 복구 대사만 소수 템플릿으로 유지한다. |
| 실행 역할 | 외부 API는 검색과 출처 수집, 애플리케이션 코드는 스키마 검증·상태 전이·행동 확정, WebLLM은 검증된 `responseGoal + facts + forbidden`을 카루아 말투로 표현하는 역할만 맡는다. |
| 컨텍스트 원칙 | WebLLM에는 전체 대화 전문 대신 최근 대화 요약, 현재 상태, 이번 응답 목표와 검증된 사실만 전달한다. 실패하거나 시간 예산을 넘으면 기본 템플릿을 즉시 사용한다. |
| 미등록 칵테일 흐름 | 이름만 `discovered` 상태로 저장하고 대화를 이어간다. 외부 API 또는 서버 측 AI가 IBA 등 출처를 탐색해 설명·레시피 후보를 보강하며, 검수 전에는 정식 추천 후보로 승격하지 않는다. |
| 저장 경계 | `discovered`, `verified`, `official` 상태와 `recommendationEnabled`, 출처 URL, 조회 시각을 분리한다. 이름과 임시 설명만 있는 항목은 도감 또는 발견 기록에는 표시할 수 있지만 추천 결정에는 사용하지 않는다. |
| LLM 경계 | WebLLM은 오프라인 임시 설명이나 카루아 말투 포장에 사용할 수 있다. 외부 API 기반 AI는 검색 결과 정규화와 요약 후보 생성에 사용할 수 있으나, 출처 사실과 정식 DB 승격은 애플리케이션 검증 계약을 통과해야 한다. |
| 미결정 사항 | 외부 검색 API, 서버 유무, 사용자별 발견 기록 저장소, 자동 승격 허용 여부, 출처별 신뢰도 정책 |
| 승인 조건 | JSON 스키마, 저장 경계, 실패 복구, 개인정보·비용 정책, 출처 검증 절차를 별도 설계하고 테스트 가능한 완료 조건을 확정할 것 |

## DEC-018: 추천 질문 축 간소화

| 항목 | 내용 |
|---|---|
| 결정 내용 | 단맛과 산미 질문을 `맛과 풍미` 질문 하나로 통합하고, 도수·탄산·베이스 질문을 유지한다. 첫 질문은 전문 용어보다 이해하기 쉬운 맛과 풍미를 우선한다. |
| 제외 항목 | 롱·숏 드링크는 현재 DB에 독립 필드가 없고 도수·탄산과 의미가 겹치므로 별도 질문으로 사용하지 않는다. |
| 비교 결과 | 기존 5축은 180조합 중 정확 일치 74개였고, 새 4축은 120조합 중 정확 일치 70개다. 최근접 복구는 106개에서 50개로 감소한다. |
| 이유 | 질문 수와 설문 느낌을 줄이면서 사용자가 체감하기 쉬운 취향을 먼저 묻고, 현재 DB가 실제로 표현할 수 있는 값만 추천 계약으로 사용하기 위해서다. |
| 재검토 조건 | 모든 칵테일에 검수된 `serving_style`과 질감 속성이 추가되고 롱·숏 선호가 실제 추천 품질을 개선한다는 근거가 생길 때 |

## DEC-017: 추천 카드와 카루아 대화 문구 분리

| 항목 | 내용 |
|---|---|
| 결정 내용 | 추천 카드는 기존의 이름, 이미지, 분위기 요약, 설명, 베이스, 재료, 잔, 분류, 맛 프로필을 유지한다. 문장형 설명은 DB 기반 중립 문구를 사용하며, 카루아식 추천 멘트와 구조화 추천 이유의 표현은 대화창에만 남긴다. |
| 이유 | 카드의 정보량을 유지하면서 정보 카드와 캐릭터 발화를 구분하기 위해서다. |
| 금지 | 카드의 설명과 상세 정보에 카루아 말투, 농담, 추천 이유 문장을 섞는 것 |
| 장점 | 상세 정보 접근성을 유지하면서 캐릭터 말투의 위치가 일관된다. |
| 단점 | 카드 정보량이 많아 모바일 레이아웃 검수가 필요하다. |
| 되돌릴 조건 | 카드 상세 정보가 모바일 핵심 흐름을 반복적으로 방해할 때 |

## DEC-015: WebLLM 말투 포장 전용 및 잠정 보류

| 항목 | 내용 |
|---|---|
| 결정 내용 | WebLLM은 JSON·DB·규칙 엔진이 내용과 판단을 모두 확정한 답안을 입력받아 카루아 말투로 포장하는 선택적 후처리 계층으로만 사용한다. 현재 구현과 연결 작업은 잠정 보류한다. |
| 자체 로직 책임 | 사용자 입력 해석, 키워드 파싱, 맥락 상태, 질문 선택, 추천 결과, 추천 근거, 설명·레시피·안전 응답의 원본 내용 확정 |
| WebLLM 허용 | 확정된 원본 답안의 의미와 사실을 바꾸지 않는 짧은 말투 변환 |
| WebLLM 금지 | 자유 입력 신호 추출, 상태 갱신 후보 생성, 질문·추천·사실 결정, DB 조회, 안전 응답 변환 |
| 이유 | 브라우저 멈춤과 비정상 출력 위험을 제거하고, WebLLM 없이 대화와 추천이 완성되는 구조를 먼저 안정화하기 위해서다. |
| 재개 조건 | JSON·규칙 기반 전체 대화·추천 흐름과 테스트가 완성되고, 말투 변환만으로 명확한 사용자 가치가 있으며 목표 기기 안정성 기준을 충족할 때 |
| 대체 결정 | DEC-013의 WebLLM 질문 표현 및 자유 입력 신호 추출 책임, DEC-004~006·DEC-011의 즉시 실행 계획을 잠정 보류한다. |

## DEC-013: 단계적 대화형 추천 질문

| 항목 | 내용 |
|---|---|

| 공통 계약 | 추천 상태 스키마, 허용 값, 질문 이력, 추천 종료 조건은 WebLLM 도입 전후에 동일하게 유지한다. |
| 추천 엔진 책임 | 다음 질문 주제 선택, 상태 갱신 검증, 후보 필터링, 추천 결과와 근거 결정 |
| WebLLM 책임 | 없음. 향후 재개 시에도 확정된 답안의 말투 포장만 허용한다. |
| 금지 | WebLLM이 자유 입력을 해석하거나 추천 상태, 칵테일 후보와 결과를 선택 또는 변경하는 것 |
| 이유 | WebLLM 이전에도 안정적인 추천 흐름을 제공하고, 도입 후 설문 느낌을 줄이면서 추천 재현성과 복구 경로를 유지하기 위해서다. |
| 되돌릴 조건 | DEC-015로 대체되어 해당 없음 |

## DEC-012: 시에스타 만담 이벤트

| 항목 | 내용 |
|---|---|
| 결정 내용 | 카루아를 주 대화 캐릭터로 유지하면서 시에스타가 손님-카루아 대화에 불쑥 난입해 한쪽 또는 양쪽과 짧은 만담을 나눈 뒤 바 업무로 돌아간다. |
| 역할 경계 | 시에스타는 추천 결과를 결정하거나 대화의 주도권을 장시간 가져가지 않는다. 캐릭터 선택 기능도 제공하지 않는다. |
| 기본 길이 | 한 이벤트는 일반적으로 시에스타와 카루아를 합쳐 2~4개의 짧은 발화로 끝낸다. |
| 장면 문법 | 예고 없는 난입으로 시작하고 창고 정리, 청소, 재고 확인 등 일상 업무 복귀로 끝낸다. |
| 발생 조건 | 추천 흐름을 방해하지 않는 일반 대화 구간, 추천 완료 직후, 재방문 또는 충분한 대화 교환 후 |
| 금지 조건 | 안전 확인 중, 사용자가 명시적 질문에 답하는 중, 추천 필터 진행 중, 퇴장 또는 초기화 중 |
| 이유 | 카루아의 주인공성을 유지하면서 두 사람의 관계와 Re:Station의 생활감을 짧게 보여준다. |
| 되돌릴 조건 | 이벤트가 핵심 추천 흐름을 자주 끊거나 시에스타가 카루아보다 강하게 인식될 때 |

## DEC-010: 이름 검색 우선

| 항목 | 내용 |
|---|---|
| 결정 내용 | 입력에 등록된 칵테일 이름 또는 별칭이 포함되면 추천 의도가 함께 있어도 해당 칵테일을 우선 표시한다. |
| 이유 | 사용자가 특정 칵테일을 언급한 경우 일반 취향 탐색보다 의도가 구체적이다. |
| 경계 | 이름 검색은 이름과 별칭만 매칭한다. 재료, 향, 베이스 표현은 취향 기반 추천 흐름으로 전달한다. |
