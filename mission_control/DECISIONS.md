## DEC-002: 카루아 단일 캐릭터 MVP

| 항목 | 내용 |
|---|---|
| 결정 내용 | 초기 MVP는 카루아와의 짧은 대화 및 칵테일 추천에 집중한다. |
| 이유 | 두 캐릭터와 세계관 확장을 동시에 구현하면 핵심 대화 품질과 추천 경험 검증이 늦어진다. |
| 고려한 대안 | 시에스타와 카루아 동시 구현, 캐릭터 선택 제공 |
| 장점 | 카루아 말투, 대화 흐름, 추천 UX에 집중 가능 |
| 단점 | 초기 세계관과 캐릭터 다양성이 축소됨 |
| 되돌릴 조건 | 카루아 단일 캐릭터 MVP의 핵심 흐름과 품질이 검증된 후 |

## DEC-003: 추천과 생성형 대화 분리

| 항목 | 내용 |
|---|---|
| 결정 내용 | 추천 엔진이 추천 결과와 근거를 결정하고, WebLLM은 이를 카루아식 문장으로 표현한다. |
| 이유 | 추천의 재현성, 설명 가능성, 오정보 방지가 필요하다. |
| 고려한 대안 | WebLLM이 직접 칵테일 선택 |
| 장점 | 모델 교체와 실패에 관계없이 추천 품질 유지 |
| 단점 | 추천 데이터와 근거 구조를 별도로 설계해야 함 |
| 되돌릴 조건 | 없음. MVP의 핵심 제품 계약으로 유지 |

## DEC-004: 프론트엔드 단독 WebLLM

| 항목 | 내용 |
|---|---|
| 결정 내용 | WebLLM을 Web Worker에서 실행하며 백엔드 없이 동작한다. |
| 이유 | 프론트엔드 단독 MVP 범위와 개인정보 보호, 오프라인 가능성을 유지한다. |
| 고려한 대안 | OpenAI API, Ollama 서버, 별도 Node 또는 Spring 백엔드 |
| 장점 | 서버 운영과 API 키가 필요 없음 |
| 단점 | WebGPU 의존, 큰 모델 다운로드, 기기별 성능 편차 |
| 되돌릴 조건 | 목표 기기에서 허용 가능한 품질과 성능을 달성할 수 없을 때 |

## DEC-005: Qwen 및 Gemma 평가 후 모델 선택

| 항목 | 내용 |
|---|---|
| 결정 내용 | Qwen 계열을 우선 후보, Gemma 계열을 비교 후보로 두고 실제 WebLLM 지원 모델을 평가한다. |
| 이유 | 한국어 어휘력뿐 아니라 브라우저 실행 가능성과 카루아 말투 유지 능력을 함께 검증해야 한다. |
| 고려한 대안 | 모델 하나를 사전 확정 |
| 장점 | 품질과 성능을 근거로 선택 가능 |
| 단점 | 평가 세트와 비교 작업이 추가됨 |
| 되돌릴 조건 | 후보 모델이 WebLLM에서 안정적으로 실행되지 않거나 더 적합한 후보가 확인될 때 |

## DEC-008: 캐릭터 대화 설계 계약

| 항목 | 내용 |
|---|---|
| 결정 내용 | 규칙 엔진, WebLLM 프롬프트, 대사 원고, 평가 세트는 `CHARACTER_DESIGN.md`의 발화 알고리즘과 금지 패턴을 따른다. |
| 이유 | 단순 키워드나 예시 대사만으로는 캐릭터 말투가 평균적인 바텐더 말투로 흔들릴 수 있다. |
| 고려한 대안 | 각 구현 파일에 캐릭터 설명을 개별 작성 |
| 장점 | 생성과 검수 기준이 일치하고 두 캐릭터의 차이를 유지하기 쉬움 |
| 단점 | 캐릭터 설정 변경 시 관련 평가와 프롬프트를 함께 갱신해야 함 |
| 되돌릴 조건 | 더 체계적인 캐릭터 대화 명세 시스템으로 완전히 대체할 때 |
